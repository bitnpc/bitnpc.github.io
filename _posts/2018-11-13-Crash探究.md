今天从操作系统层面来谈谈iOS中App的Crash。

# Crash是什么？
Crash本质是操作系统的一种异常控制。  

现代操作系统需要有一整套完善的处理流程来解决异常。  
在硬件层，硬件检测到的事件，会出发控制突然转移到异常处理程序。  
在操作系统层，内核通过上下文转换将控制从一个用户进程转移到另一个用户进程。  
在应用层，一个进程可以发送信号到另一个进程。如在终端执行kill命令，杀死其他进程。

操作系统的异常分为4种。异常分为4种。中断（interrupt），陷阱（trap），故障（fault）和中止（abort）。 
|-----类别-----|-----------原因------------|---异步/同步---|----------返回行为----------|
|-----中断-----|----来自I/O设备的信号---|------异步------|--总是返回到下一个指令--|
|-----陷阱-----|-------有意的异常--------|------同步------|--总是返回到下一个指令--|
|-----故障-----|----潜在可恢复的错误---|------同步------|----可能返回到当前指令----|
|-----中断-----|--0--不可恢复的错误-----|------同步------|---------不会返回----------|

App的Crash是由于进程的故障/中止。   
为了方便表述，下面将App的Crash和进程的Crash等同使用。因为iOS系统中的App只有一个进程。

# Crash的原因
App发生Crash的原因可能有很多种。  
### CPU无法执行代码
- 非法算术运算。除0。  
- 无效的指令。如基于x86_64架构的CPU(CISC指令集)，无法运行在基于ARM架构的CPU(RISC指令集)。  
- 无效的内存地址。应用程序在内存中的内存布局如下图所示。  

由低地址到高地址依次为：  
程序文本（.text）  
已初始化的数据（.data）  
未初始化的数据（.bss）  
堆（heap）  
栈（stack）  

当进程在执行时，会在栈中创建保存指针的变量pointer，pointer指向堆中对象的内存地址。如果堆中对象的内存地址被释放，但是pointer未置空，那么随着进程不断执行，后续会向这块堆中的内存写数据，如果继续访问该pointer，则很可能会导致Crash。这种pointer被称为野指针。通常情况下，多线程操作是导致这种情况出现的主要因素。  

### 操作系统出于某种因素
- WatchDog事件。用来检测应用是否长时间无响应，模拟器无此功能。
```
setWatchTimer(Uint32 timeOut)
```

- 机器过热  
可能是由于CPU负载太高。操作系统会优先kill掉CPU占有率过高的App.

- 内存不足  
App在运行期间可能会收到memoryWarning。在这个方法里面应该做一些释放内存的操作。  

- 签名无效  
比如企业证书到期，App启动时就会直接退出。  

### 编程语言的保护措施
- 数据越界。
不是所有语言都提供这种保护措施。如C语言就不提供数据越界的保护。  
如下代码，在C语言中是不会Crash的，但是打印出来的字符是未知的。  
```
int main () {
    char *str = "bitnpc";
    char c = str[666];
    printf("%c\n", c);
}
```
- 找不到指定方法。Runtime提供的保护措施。

### 开发者的防范措施
这里主要是指用Assert来检查某些重要的参数。

# Crash传递的流程
当App发生Crash时，是有一定的传递流程的。  
其顺序为Mach内核->Unix->iOS。  

- Mach异常是最底层的内核级异常，如EXC_BAD_ACCESS。其他异常可见
[exception_types](https://opensource.apple.com/source/xnu/xnu-3789.1.32/osfmk/mach/exception_types.h.auto.html)     

- Unix Signal是Unix系统的一种异步通知机制。Mach异常在host层被ux_exception转换为相应的Unix Signal，并通过threadsignal将信号投递到出错的线程。如SIGABRT，SIGSEGV。在mac OS系统中，可以输入"man signal"查看所有的Signal列表。在[这里](https://github.com/torvalds/linux/blob/master/include/linux/signal.h)也可以看到。    

- NSException是Objective-C层。由iOS系统库或者runtime，或其他第三方库验证而抛出的异常。  

##### 常见的exception_types。
|---Exception类型----|-------描述-------|-----------说明-------------|
|--EXC_BAD_ACCESS---|Bad Memory Access|错误内存地址，访问的地址不存在或者当前进程没有权限都会报这个错。|
|-----EXC_CRASH-----|--Abnormal Exit--|通常跟随的UNIX Signal是SIGABRT，当前进程被系统检测到有异常行为而杀死|
|EXC_BAD_INSTRUCTION|Illegal Instruction|非法或未定义的指令或操作数|
##### 常见的Unix信号
|--Unix Signal---|------------------说明-------------------|
|-----SIGSEGV----|访问了无效的内存地址，这个地址存在，但是当前进程没有权限访问它。属于硬件层错误|
|-----SIGABRT----|程序Crash，这个符号是由C函数abort()触发的。通常代表系统发现当前程序执行出错。属于软件层错误|
|-----SIGBUS-----|访问了无效的内存地址，与SIGSEGV的区别是：SIGBUS表示内存地址不存在。属于硬件层错误|
|-----SIGTRAP----|Debugger相关|
|-----SIGILL-----|尝试执行一个非法的、未知的、没有权限的指令|

# Crash堆栈捕获与符号化
上面说到了Crash传递的流程，所以捕获的时候，就可以分别在对应的层次捕获堆栈信息。  
iOS上有一个著名的Crash捕获的库，叫做PLCrashReporter。  

它在Mach层面和Unix层面都做了一些处理工作。   

#### Mach层
在Mach层面，捕获异常的操作。  
大概流程是初始化上下文，初始化端口与权限，把port加入到port set中，以接受回调。循环等待消息。
具体代码可查看[PLCrashMachExceptionServer](https://github.com/plausiblelabs/plcrashreporter/blob/master/Source/PLCrashMachExceptionServer.m)
```
- (id) initWithCallBack: (PLCrashMachExceptionHandlerCallback) callback
                context: (void *) context
                error: (NSError **) outError
{
    // Initialize the bare context. 
    // Initalize our server's port
    // Initialize our notification port
    // Initialize our port set.
    // Add the service port to the port set
    // Add the notify port to the port set
    // Spawn the server thread.
}
```

#### Unix层
在Unix层则可以注册Signal处理回调。如下代码是把接受到的SIGSEGV信号统一用signalHander来处理。
```
Signal(SIGSEGV, signalHander);   
```

Crash捕获后获得的数据都是对应的虚拟内存地址。为了排查Crash的原因，需要对其符号化，使之可读。  
符号化的本质是在一个映射文件中，找到内存地址对应的函数方法。   
主要有以下三种符号化方法。  
- Symbolicatecrash
- mac下的atos工具和Linux平台的替代品atosl
- 通过dSYM文件提取地址和符号的对应关系，进行符号化

系统库的符号文件，可以从iOS固件中获取，也可以从Github上开源项目中找到对应系统的符号文件。
